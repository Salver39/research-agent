## Выполненные исправления

### Этап 1 — Fix #3: Авторизация сессий (owner-token)

**Проблема:** Любой, кто знает UUID сессии, мог читать/писать чужие данные. Достаточно было подсмотреть URL.

**Решение:** Система owner-token авторизации.

**Бэкенд:**

- **`backend/db/models.py`** — добавлено поле `owner_token: Mapped[str]` в модель Session
- **`backend/api/deps.py`** — новый файл. Центральная зависимость `require_owner()`, которая:
  - Извлекает токен из заголовка `X-Owner-Token` или query-параметра `?token=`
  - Проверяет совпадение с `owner_token` в БД
  - Возвращает 401/403/404 при ошибках
- **`backend/api/session.py`** — `create_session` генерирует `owner_token`, возвращает его в ответе. `get_session` использует `require_owner`
- **`backend/api/stream.py`** — `stream_agent`, `advance_stage`, `retreat_stage` защищены через `require_owner`
- **`backend/api/upload.py`** — `upload_file` защищён через `require_owner`
- **`backend/api/download.py`** — `download_document` защищён через `require_owner`

**Фронтенд:**

- **`frontend/lib/auth.ts`** — новый файл. Утилиты:
  - `saveOwnerToken(sessionId, token)` — сохраняет в localStorage
  - `getOwnerToken(sessionId)` — читает из localStorage
  - `authHeaders(sessionId)` — возвращает `{ "X-Owner-Token": "..." }`
  - `withTokenQuery(url, sessionId)` — добавляет `?token=...` к URL (для `<a href>`)
- **`frontend/store/session.ts`** — добавлены `owner_token` и `setOwnerToken`
- **`frontend/app/page.tsx`** — вызов `saveOwnerToken` после создания сессии
- **`frontend/app/session/[id]/page.tsx`** — `authHeaders(id)` во всех fetch-запросах
- **`frontend/hooks/useSSE.ts`** — `authHeaders(sessionId)` в заголовках стрима
- **`frontend/components/wizard/ContextUploadScreen.tsx`** — `authHeaders` для загрузки файлов
- **`frontend/components/wizard/OutputScreen.tsx`** — `authHeaders` для генерации, `withTokenQuery` для ссылок на скачивание

### Этап 2 — Fix #4 + #10: Race condition в stream/advance/upload

**Проблема:** Параллельные запросы (stream + upload, advance + stream) могли перезаписать state друг друга — последний коммит затирал изменения первого.

**Решение:** Per-session in-memory asyncio locks + merge-write паттерн.

- **`backend/api/locks.py`** — новый файл. `session_lock(session_id)` — async context manager, выдаёт per-session asyncio.Lock. Реестр замков защищён отдельным `_registry_lock`.
- **`backend/api/stream.py`** — переписан:
  - Snapshot state под замком перед стримом
  - Стрим выполняется БЕЗ удержания замка (чтобы upload/advance могли работать параллельно)
  - Merge-write в конце стрима: под замком перечитывает свежий state из БД, накладывает только stream-owned ключи (`_STREAM_OWNED_KEYS`, `_STREAM_OWNED_CONTEXT_KEYS`)
  - Новая функция `_merge_stream_state(fresh, after_stream)` — overlay только своих ключей
- **`backend/api/upload.py`** — pre-check бюджета под замком, запись source-записи под замком с double-check
- **`backend/api/stream.py`** — `advance_stage` и `retreat_stage` используют `session_lock` + `SessionLocal()` для свежего read-modify-write

### Этап 4 — Fix #6: Лимиты на загрузку файлов

**Проблема:** Эндпоинт `/upload` не ограничивал размер файлов. Можно было загрузить гигабайтный файл и забить диск.

**Решение:** Chunked streaming upload с лимитами.

- **`backend/api/upload.py`** — полная переработка:
  - Константы: `MAX_FILE_BYTES = 25 MB`, `MAX_SESSION_BYTES = 50 MB`, `CHUNK_SIZE = 64 KB`
  - `ALLOWED_EXTENSIONS = {".pdf", ".docx", ".txt", ".csv"}` — валидация расширений
  - `_stream_to_disk(file, dest, max_bytes)` — стримит файл на диск чанками по 64KB, при превышении лимита удаляет файл и возвращает 413
  - `_unique_dest(dirpath, filename)` — коллизии имён через суффиксы `_2`, `_3`
  - `_safe_unlink(path)` — безопасное удаление файла
  - Pre-check бюджета сессии под замком
  - Double-check при записи source-записи (защита от concurrent upload)
  - Source-записи теперь включают `size` и `original_name`

### Этап 5 — Fix #7: PDF молча возвращал DOCX (Вариант A)

**Проблема:** Если weasyprint/mammoth не могли сконвертировать в PDF, `_save()` молча возвращал путь к .docx файлу. Пользователь думал, что скачивает PDF, но получал DOCX.

**Решение:** Честные ошибки вместо молчаливого fallback.

- **`backend/documents/generator.py`**:
  - `_save()` возвращает `None` при ошибке PDF-конвертации (вместо fallback на docx)
  - `generate_all()` в ZIP-режиме фильтрует `None`-результаты
- **`backend/api/download.py`**:
  - Если `format=pdf` и результат `None` — возвращает HTTP 503 "PDF generation unavailable — download .docx instead"
- **`frontend/components/wizard/OutputScreen.tsx`**:
  - PDF-кнопка через `fetch` вместо `<a href>` — перехватывает 503
  - Показывает amber-баннер "PDF-генерация недоступна. Скачайте .docx."
  - DOCX-ссылки остаются обычными `<a>`

### Этап 6 — Fix #8: Двойная генерация ZIP

**Проблема:** Кнопка "Сгенерировать все документы" делала fetch к `/download?format=zip`, но blob отбрасывался. Затем ссылка "Скачать всё (ZIP)" делала второй запрос — полная повторная генерация.

**Решение:**

- **`frontend/components/wizard/OutputScreen.tsx`**:
  - Добавлен state `zipUrl`
  - `handleGenerate()` сохраняет blob через `URL.createObjectURL(blob)`
  - Ссылка "Скачать всё (ZIP)" использует сохранённый `zipUrl` вместо нового запроса
  - Добавлен атрибут `download="research_package.zip"`

### Этап 7 — Fix #9: Двойной onDone в SSE

**Проблема:** Если `[DONE]` приходил в конце буферизированного чанка, парсер мог вызвать `onDone` дважды (один раз при парсинге строк, второй раз при обработке остатка буфера). `[ERROR]` в остатке буфера не обрабатывался.

**Решение:**

- **`frontend/hooks/useSSE.ts`**:
  - Добавлен флаг `doneFired` — гарантирует однократный вызов `onDone`
  - Вынесена функция `finish()` — единая точка завершения с проверкой флага
  - Вынесена функция `processLine(line)` — единообразная обработка SSE-строк (data, [DONE], [ERROR])
  - Блок `if (done)` теперь обрабатывает остаток буфера через `processLine` и вызывает `finish()` — без дублирования

## Дополнительный фикс: совместимость с Python 3.9

**Проблема:** Бэкенд работает на Python 3.9.6, а в `session.py` использовался синтаксис `dict | None` (PEP 604, Python 3.10+). При авто-перезагрузке uvicorn падал на импорте — все запросы зависали, пользователь получал HTTP 500 при создании сессии.

**Решение:**

- **`backend/api/session.py`**:
  - Добавлен `from __future__ import annotations`
  - `dict | None` → `Optional[dict]`, `list | None` → `Optional[list]`
  - Добавлен `from typing import Optional`

## Обнаруженная, но не исправленная проблема

### Медленная генерация дизайна исследования (design stage)

**Симптом:** Стадия "Дизайн исследования" зависает на 2-5+ минут.

**Причина:** `DesignUsabilityAgent` и `DesignInterviewsAgent` используют `asyncio.gather()` для параллельных non-streaming вызовов к OpenAI:
- 1 вызов на каркас (2500 токенов)
- N вызовов на гипотезы (1200 токенов каждый)

Проблемы:
1. `complete()` ждёт полный ответ без стриминга (30-60с на вызов)
2. OpenAI может троттлить параллельные запросы по RPM/TPM
3. Результат отдаётся одним куском только после завершения ВСЕХ вызовов
4. Фронтенд обрывает SSE через 240с, но UI не всегда корректно это отображает

**Рекомендованное решение:** Заменить N параллельных non-streaming вызовов на 1 streaming-вызов (как в ChatGPT) — все гипотезы в промпте, ответ стримится пользователю по мере генерации. Планируется в следующей итерации.
