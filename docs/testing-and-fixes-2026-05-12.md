## 2. Диагностика проблем с OpenAI API

### 2.1 Первый запуск — `insufficient_quota` (429)

При генерации диагноза агент получал от OpenAI ошибку **429 insufficient_quota**. Это **не баг в коде** — у аккаунта закончилась квота / не было биллинга.

После пополнения счёта пользователем — ошибка исчезла.

### 2.2 Уточнение про gpt-5.5

В диалоге выяснилось:
- Модель **gpt-5.5 реально существует** (релиз OpenAI — апрель 2026)
- Мой knowledge cutoff январь 2026 — не знал про неё
- В `.env` стояла корректная модель, я ошибочно подозревал опечатку

### 2.3 Вторая проблема — `rate_limit_exceeded` на этапе дизайна

```
Rate limit reached for gpt-5.5 in organization org-...
on tokens per min (TPM): Limit 500000, Used 500000
code: rate_limit_exceeded
```

**Это не quota (биллинг), а TPM (rate limit).** Совсем другая категория ошибки.

**Причина:** на этапе «Дизайн исследования» три design-агента запускают через `asyncio.gather` параллельные запросы к OpenAI (1 frame + по 1 на каждую гипотезу). При 10–15 гипотезах = 11–16 одновременных вызовов → бьётся об TPM-лимит даже на пополненном аккаунте.

## 3. Реализация решения: `asyncio.Semaphore`

### Изменения в `backend/agents/base.py`

Добавлено:
1. `import asyncio` в импорты
2. Модульный `_OPENAI_SEMAPHORE: asyncio.Semaphore | None = None`
3. Функция `_get_semaphore()` — lazy init с чтением `OPENAI_MAX_CONCURRENCY` из env (default = 3)
4. `complete()` обёрнут в `async with _get_semaphore():` — теперь не более N параллельных вызовов одновременно

### Изменения в `backend/.env.example`

Добавлена строка с комментарием:
```
OPENAI_MAX_CONCURRENCY=3
```

### Изменения в `backend/.env`

Также добавлена та же переменная (изначально 3, потом для диагностики поднята до 20).

### Качество результата (важно)

Семафор **не влияет на качество выдачи агента**:
- Каждый вызов OpenAI stateless и независим
- Промпт, контекст, max_tokens, temperature — те же
- Меняется только скорость отправки запросов

## 4. Обнаружение второй, более глубокой проблемы — пост-burst hang

После реализации семафора при тестировании выяснилось:
- С `concurrency=3` дизайн висит **>6 минут**
- С `concurrency=20` (фактически без ограничений) — все запросы улетают разом, **15 успешных ответов** от OpenAI приходят пачкой за ~13 секунд
- Но **после** прихода всех ответов backend **всё равно зависает** — `[DONE]` фронту не уходит
- Через 8 минут frontend сам прерывает по AbortController → красный баннер «Превышено время ожидания»

### Гипотезы (не подтверждены)

Зависание происходит между `asyncio.gather` и финальным `yield "[DONE]"`. Возможные причины:
1. `_persist_agent_output` — JSON-парсинг ответов модели
2. Запись в SQLite под `session_lock`
3. Какая-то корутина в gather не завершилась (несмотря на видимые 15 HTTP-ответов)
4. Python 3.9 + asyncio + httpx edge case (известны проблемы с gather на Python < 3.11)

### Почему точно не диагностировано

После чистки логов (`cleanup-2026-05-11.md`, пункт 7) `logger.info` понижены до `debug`. **Именно те логи, что нужны** для диагностики post-burst hang, теперь молчат. Это моя ошибка в предыдущей сессии — суммарно по фиче «чистка логов» получилось перестраховаться.

Для полноценной диагностики нужно временно вернуть INFO-уровень в `stream.py`. **Не сделано** в этой сессии.

## 5. Прагматичное решение — переключение на gpt-4o

### Изменения в `backend/.env`

```
OPENAI_MODEL=gpt-5.5      → gpt-4o
OPENAI_MODEL_MINI=gpt-5.4 → gpt-4o-mini
```

### Результат

- **Wizard работает end-to-end.** Дизайн генерируется за 20–60 секунд (вместо 4+ минут на gpt-5.5).
- Все этапы — диагноз, бриф, гипотезы, метод, выборка, дизайн, документы — успешно проходят.
- Post-burst hang **не воспроизводится** на gpt-4o (возможно, был связан с timing/retry edge case у gpt-5.5).

### Открытая претензия пользователя

**Качество gpt-4o ниже, чем у gpt-5.5.** Это ожидаемо: gpt-5.5 как более новая модель сильнее на сложных промптах с многошаговым рассуждением.

Возможные направления (не реализованы в сессии):
- Гибридная конфигурация: gpt-5.5 везде, gpt-4o только в design-агентах
- Тюнинг промптов под gpt-4o
- Диагностика и устранение post-burst hang → возврат на gpt-5.5 полностью

## 6. Разъяснение про папку `examples/`

При тестировании пользователь заметил, что агент «не читал» документы, которые он раньше загружал — после того как мы перенесли их в `examples/`.

### Что в `examples/` на самом деле

В `cleanup-2026-05-11.md`, пункт 1, в эту папку были скопированы **6 готовых `.docx`** — это **выходные артефакты** одной из предыдущих тестовых сессий (Discussion Guide, Скринер, Брифинг и т.д.). Папка предназначена **для демонстрации ревьюеру GitLab**, не для агента.

### Куда уходят документы, которые пользователь грузит в wizard

```
Браузер (экран «Данные компании»)
      ↓ POST /api/upload/{session_id}
backend/uploads/{session_id}/<имя файла>
      ↓ rag/indexer.py
backend/chroma_db/ (векторный индекс session-specific)
      ↓ на этапе Context Agent
rag/retriever.py — топ-6 фрагментов по семантической близости к research_question
      ↓
встраиваются в промпт ContextAgent
```

**`examples/` агент не читает и не должен — это не входные данные.**

### Почему пользователь мог решить, что «агент не прочитал»

Версии:
1. Документы не загрузились до этапа контекста (пропустил шаг)
2. RAG не нашёл семантической связи между research_question и документами
3. ContextAgent **не настроен** на style transfer — он извлекает паттерны, но не следует формату примеров
