# План публичного запуска

**Цель:** показать экспертизу в UX-исследованиях, получить лиды на консалтинг/заказы. Не продукт, не SaaS.

**ЦА:** фрилансеры и стартапы (продакты, дизайнеры, исследователи). Не корпорации.

**Сетап:** два пути доступа на одном лендинге.
1. **Веб-демо** — одна бесплатная сессия на email. Основной путь, низкий барьер.
2. **Self-host через docker-compose** — для тех, кто работает с чувствительными данными или хочет прогнать на своём ключе.

---

## Экономика (подтверждено по данным 11–17.05)

| Метрика | Значение |
|---|---|
| Средняя стоимость сессии | ~$0.91 |
| Худшая наблюдённая | $4.47 |
| Embeddings (Chroma локально) | $0 — навсегда |
| ~17 вызовов модели на одну сессию | (важно для kill-switch) |
| Месячный потолок при `$30/день` kill-switch | $300–500 худший случай, $100–200 реалистично |

Решения, которые следуют из экономики:
- **BYOK не нужен на старте.** Цена низкая, лимит «1 сессия / email» защищает от abuse.
- **Self-host бесплатен для тебя** — embeddings локальные, completions платит юзер своим ключом.

## Фаза 1. Веб-демо (1–2 недели)

### Что добавить в код

| Задача | Где | Оценка |
|---|---|---|
| Magic-link auth (email → токен) | новый модуль `api/auth.py`, использовать Resend/Loops для отправки | 1 день |
| Поле `email` и связь с сессиями | `db/models.py` — таблица `User`, FK на `Session` | 0.5 дня |
| Гейт «1 завершённая сессия на email» | в `api/session.py` при создании сессии | 0.5 дня |
| Cloudflare Turnstile на форму signup | фронт + проверка на бэке | 0.5 дня |
| Kill-switch по дневному бюджету | cron-задача или middleware, читающая OpenAI Usage API | 0.5 дня |
| Persistent volume для SQLite на Railway | конфиг Railway, путь `DATABASE_URL` | 0.5 дня |
| Cap на длину user input (бриф ≤ 20k символов) | фронт + валидация в `api/session.py` | 0.5 дня |
| Privacy-disclaimer в UI перед загрузкой файла | компонент в `frontend/components/wizard/` | 0.5 дня |
| Обновить Next.js до актуальной версии | `frontend/package.json` + smoke-test | 0.5–1 день |

### Деплой

- **Frontend → Vercel** (Next.js, проще некуда).
- **Backend → Railway** (Dockerfile с системными deps для WeasyPrint, persistent volume для SQLite + `uploads/` + `chroma_db/` + `outputs/`).
- **Не использовать Vercel для бэка** — SSE-стримы у тебя длинные (минуты), функции Vercel прибьют по таймауту.

### Чего НЕ делать в Фазе 1

- BYOK интерфейс.
- Подписку / Stripe.
- Azure OpenAI / ZDR.
- Postgres миграцию (SQLite на persistent volume справится).
- S3/R2 для файлов (Railway volume хватит на 1–2 тысячи сессий).
- SSO, audit log, ролевую модель.

## Фаза 2. Self-host через docker-compose (3–5 дней)

### Что сделать

| Задача | Файлы | Оценка |
|---|---|---|
| `Dockerfile` для бэка с системными deps WeasyPrint (`pango`, `cairo`, `gdk-pixbuf`, `libffi`) | `backend/Dockerfile` | 1 день |
| `Dockerfile` для фронта (multi-stage, `next build` + `next start`) | `frontend/Dockerfile` | 0.5 дня |
| `docker-compose.yml` с двумя сервисами + volume для `chroma_db`, `uploads`, `outputs`, `research_agent.db` | корень репо | 0.5 дня |
| `.env.example` с понятными комментариями: где взять OpenAI ключ, какие модели прописать | корень репо | 0.5 дня |
| Раздел «Self-host» в README с пошаговой инструкцией | `README.md` | 0.5 дня |

### Проверочный сценарий

После `docker-compose up`:
1. Открыть `http://localhost:3000` — рабочий UI.
2. Создать сессию, пройти один этап.
3. Перезапустить контейнеры — сессия осталась.

Если эти три шага проходят — self-host готов.

### Чего НЕ делать в Фазе 2

- Не делать portable bundle (`PyInstaller` + двойной клик). Docker-compose покрывает ЦА «фрилансеры/стартапы с минимальной техникой» достаточно.
- Не делать Electron / Tauri.
- Не делать install-скрипт через `curl | bash`.

## Фаза 3. Витрина (1 неделя)

### Лендинг (одна страница)

Структура сверху вниз:
1. **Hero:** заголовок «AI-помощник для подготовки UX-исследований» + кнопка «Try demo».
2. **Проблема:** 2–3 предложения о том, какую боль исследователя решает.
3. **Как работает:** 6 этапов wizard'а с GIF/скринами агентов.
4. **Demo CTA:** кнопка «Запустить бесплатную сессию» + примечание «1 бесплатный прогон / email».
5. **Privacy:** что хранится локально (только при self-host), что уходит в OpenAI. Один абзац.
6. **Self-host:** ссылка на GitHub + одна команда `docker-compose up`.
7. **Why this way:** короткий блок «почему именно такая архитектура агентов» — это и есть демонстрация экспертизы.
8. **About:** кто я, контакты для заказов (email/Telegram).

### Видео-демо (2–3 минуты)

Не монтировать долго. Сценарий:
- 0:00–0:20 — кто ты и какую задачу решает инструмент.
- 0:20–2:00 — прогон одной полной сессии на синтетическом примере.
- 2:00–2:30 — что получает пользователь на выходе (документ), как использовать дальше.
- 2:30–3:00 — где найти, как развернуть у себя, контакты.

Снимать экран + голос. Сырое, естественное — работает лучше глянцевого.

### GitHub README (перепиши)

Сейчас README — про разработчиков. Перепиши под аудиторию «продакт смотрит проект из любопытства»:
1. Что делает (1 абзац).
2. Скриншоты (3–4 штуки, ключевые экраны).
3. **Архитектурная диаграмма** (Mermaid в md или Excalidraw, картинкой) — агенты, RAG, генерация документов.
4. Ссылка на live demo.
5. Раздел «Design decisions» — какие исследовательские принципы зашиты в агентов, какие альтернативы рассматривал. Это самый ценный раздел для репутации.
6. Self-host раздел (одна команда + ссылка на полную инструкцию).
7. Контакты.

## Фаза 4. Публикация (1–2 дня)

### Где публиковать

Где сидит ЦА «продакты / дизайнеры / исследователи»:
- **vc.ru** или **Хабр** — длинная статья.
- **LinkedIn** — короткий пост со ссылкой на статью.
- **2–3 профильных Telegram-канала** про продакт-менеджмент, UX-исследования.

### Формат поста

Не «я сделал штуку». Структура **«методология → инструмент как иллюстрация»:**

1. Проблема исследователя (с примером из практики).
2. Какие исследовательские принципы зашил в агентов (это твоя экспертиза, главная часть).
3. Почему агенты, а не один большой промпт. Trade-offs.
4. Демо: ссылка + GIF.
5. Self-host для тех, у кого NDA: GitHub.
6. «Если делаешь похожее или нужна помощь — пишите».

## Что сознательно отложено (вне scope)

- BYOK интерфейс — добавлять, только если юзеры начнут просить «больше сессий».
- Подписка / Stripe — цель не выручка.
- Portable bundle (PyInstaller / Electron) — слишком много работы под цель.
- Azure OpenAI / ZDR — для enterprise, не для фрилансеров.
- Postgres миграция — SQLite справится на этом масштабе.
- Оптимизация ~17 вызовов модели на сессию — преждевременно при $0.91 за прогон.
- Свой scoring/analytics — на лендинге достаточно стандартного Plausible/Umami.

## Порядок выполнения

Минимум зависимостей между фазами, но логично делать в таком порядке:

1. **Фаза 2 (docker-compose, 3–5 дней)** — параллельно валидирует, что код вообще пакуется, и даёт self-host для GitHub раньше.
2. **Фаза 1 (веб-демо, 1–2 недели)** — основная работа.
3. **Фаза 3 (витрина, 1 неделя)** — параллельно с концом Фазы 1.
4. **Фаза 4 (публикация, 1–2 дня)** — после того, как live demo стабильно работает 3–5 дней без падений.

**Суммарно: 3–4 недели до первой публикации.**

Оценки даны как порядок величины — на твоём опыте и темпе могут отклоняться в 1.5–2 раза.
