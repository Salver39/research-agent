# Методичка — One-Pager

Короткая версия для быстрой загрузки в контекст. Полная: `docs/methodology.md`.

## 6 принципов
1. **Корень раньше симптома** — задать «что изменилось между рабочим и сломанным» ДО фикса.
2. **Foundation — день 0** — auth, `.gitignore`, fail-fast, smoke-test ДО первой фичи.
3. **Вертикальный срез** — один полный путь end-to-end, потом расширение.
4. **Измеряй раньше оптимизации** — нет цифры → нет оптимизации.
5. **Скрывай нерабочее** — никогда не stub в UI; либо работает, либо disabled.
6. **Отсрочка foundation × N итераций** — фундамент не «потом», иначе долг умножается.

## Foundation Day (день 0, обязательно)
- `.gitignore` (env, node_modules, venv, БД, кэши сборки)
- Секреты только в `.env`, шаблон в `.env.example`
- Auth / ownership даже минимальный (`owner_token` подойдёт)
- Схема данных с явным механизмом мутации/миграции
- Стратегия логгирования (уровни, категории) зафиксирована
- Fail-fast на обязательные env-переменные
- Telemetry usage/cost с первой строчки
- 1 e2e smoke-test критичного пути
- Rewind/seed утилиты для отката этапов
- ADR template (`docs/adr/`)
- `BACKLOG.md` с приоритетами P0/P1/P2

## Цикл итерации
1. Перечитать open questions прошлой итерации
2. Выбрать тему из `BACKLOG.md` с явным приоритетом
3. Если фикс — задать «что изменилось» ДО правки
4. Нетривиальное решение — ADR (1 страница)
5. Реализовать узко, без попутного рефакторинга
6. Прогнать smoke-test
7. Ретроспектива в `iteration-N.md`, апдейт `BACKLOG.md`

## Постоянные практики
- **Economics check** — раз в неделю
- **Quality eval-set** — 10 кейсов с rubric, прогон при смене модели
- **External user** — минимум 1 со стороны в неделю
- **Backlog grooming** — раз в 2 итерации

## Самопроверка
**Stop-hook (после каждого содержательного действия):**
1. Гипотеза проверена или просто наследована?
2. Лечу симптом или ищу причину?
3. Поведение резко поменялось при смене параметра? — виновата переменная, не код вокруг.
4. Задал «что изменилось между рабочим и сломанным» в первую очередь?

**В начале итерации:**
5. Перечитал прошлую ретроспективу?
6. Делаю по приоритету или «что бросилось в глаза»?
7. Есть простой smoke-тест для этой правки?

## Anti-patterns (топ-5)
1. **Cleanup как отдельная итерация** → знак, что foundation не сделан день 0.
2. **Stub в UI, который пользователь видит** → скрыть/disabled, не оставлять.
3. **Silent fallback на модель/конфиг** → fail-fast.
4. **Singleton/глобал без ADR** → переоткроется через 3 итерации.
5. **Наследование гипотезы без проверки** → 5 итераций лечения симптомов.

## Правила решений (короткие)
- **Refactor компонента:** >200 строк И правки в каждой из 3 последних итераций → сейчас. Иначе — отложить.
- **Cut или hide:** не успевает к релизу или работает <80% → hide, не stub.
- **Смена модели/версии/зависимости:** замер ДО → замер ПОСЛЕ. Без замеров не меняем.
- **Премат-опт (три вопроса):** измерил? простой вариант пробовал? есть метрика «достаточно»? — нет хотя бы одного «да» → не оптимизируем.

## Что нужно отдельно (не входит в методичку)
PRD, vision, road-map, user stories, ADR — продуктовые/инженерные документы, существование которых методичка предполагает.

## Полная версия
`docs/methodology.md` — развёрнутые объяснения, оговорки, примеры из реального проекта (Research Preparation Agent, iter 1–13).
