## 5. Решения по архитектурным изменениям

В этой сессии **сознательно отказались** от ряда крупных рефакторов из-за принципа «стабильность для пользователя > эстетика кода»:

| Что не делали | Почему |
|---|---|
| Унификация трёх Design-агентов в `BaseParallelDesignAgent` | Критичный user-path, нет автотестов, риск регрессии |
| Общий `_build_research_context()` | Изменение порядка/формата строк в промптах может сдвинуть качество LLM-ответов |
| Разбиение `DesignScreen.tsx` (365 строк) | Сложный условный рендер, регрессия = пользователь не увидит результат |
| Разбиение `HypothesesScreen.tsx` (375 строк) | Сложный streaming-state с парсингом partial JSON |
| Разбиение `session/[id]/page.tsx` (459 строк) | Замкнутая система refs + SSE handlers, центральная точка |
| Разбиение `documents/generator.py` (423 строки) | Финальный артефакт пользователя — нельзя сломать |

В `orchestrator.py` добавлен комментарий рядом с `_get_design_agent()` — для ревьюера, чтобы он понимал, что три отдельных design-агента — **осознанное решение**, не недосмотр.

## 10. Тестирование — rate limit и семафор

### Найдена ошибка

```
Rate limit reached for gpt-5.5 in organization org-...
on tokens per min (TPM): Limit 500000, Used 500000
code: rate_limit_exceeded
```

**Причина:** на этапе «Дизайн исследования» агенты запускают через `asyncio.gather` параллельные запросы (1 frame + по 1 на каждую гипотезу). При 10–15 гипотезах = 11–16 одновременных вызовов → бьётся об TPM-лимит.

### Реализован семафор

Изменения в `backend/agents/base.py`:

1. Добавлен `import asyncio`
2. Модульный `_OPENAI_SEMAPHORE: asyncio.Semaphore | None = None`
3. `_get_semaphore()` — lazy init, читает `OPENAI_MAX_CONCURRENCY` из env (default = 3)
4. `complete()` обёрнут в `async with _get_semaphore():`

Изменения в `backend/.env.example`:
```
OPENAI_MAX_CONCURRENCY=3
```

Изменения в `backend/.env`:
```
OPENAI_MAX_CONCURRENCY=20  # позже подняли для диагностики
```

Семафор **не влияет на качество выдачи** — он только лимитирует одновременные HTTP-запросы.

## 11. Тестирование — post-burst hang

После реализации семафора при тестировании:

- С `concurrency=3` дизайн висит **>6 минут**
- С `concurrency=20` все 11 запросов улетают разом → 15 успешных ответов (с учётом retry SDK) приходят пачкой за ~13 секунд
- **После прихода всех ответов backend всё равно зависает** — `[DONE]` фронту не уходит
- Через 8 минут frontend сам прерывает по AbortController → красный баннер «Превышено время ожидания»

### Гипотезы (не подтверждены)

Зависание происходит между `asyncio.gather` и финальным `yield "[DONE]"`. Возможные кандидаты:

1. `_persist_agent_output` — JSON-парсинг ответов модели
2. Запись в SQLite под `session_lock`
3. Корутина в `gather` не завершилась
4. Python 3.9 + asyncio + httpx edge case (Python 3.11+ имеет несколько фиксов на эту тему)

### Почему точно не диагностировано

На этапе 8 мы сами понизили `logger.info` → `debug` в `stream.py`. **Именно те логи, что нужны** для диагностики post-burst hang, теперь молчат. Для полноценной отладки нужно временно вернуть INFO-уровень. **Не сделано** в этой сессии.

## 13. Разъяснение про папку examples

При тестировании пользователь подумал, что его загруженные документы оказались в `examples/` и агент «не читал» их.

### Что в `examples/` на самом деле

6 готовых `.docx` — это **выходные артефакты** одной из предыдущих тестовых сессий (Discussion Guide, Скринер, Брифинг, и т.д.). Папка нужна **для демонстрации ревьюеру GitLab**: «вот какой пакет документов система генерирует». Агент туда **не смотрит** и не должен.

### Куда уходят документы пользователя

```
Браузер (экран «Данные компании»)
      ↓ POST /api/upload/{session_id}
backend/uploads/{session_id}/<имя файла>
      ↓ rag/indexer.py — чанки + эмбеддинги
backend/chroma_db/ — векторный индекс session-specific
      ↓ на этапе Context Agent
rag/retriever.py — топ-6 фрагментов по близости к research_question
      ↓
встраиваются в промпт ContextAgent
```

### Почему пользователь решил, что «агент не прочитал»

Возможные причины:

1. Документы не загрузились до этапа контекста (пропустил шаг)
2. RAG не нашёл семантической связи между research_question и документами
3. ContextAgent **не настроен** на style transfer — он извлекает паттерны, но не следует формату примеров. Это отдельная задача, требует переработки промпта `ContextAgent`.
